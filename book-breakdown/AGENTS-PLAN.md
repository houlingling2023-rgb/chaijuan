# 拆卷 · Agent 架构方案

---

## 架构总览

```
灵儿（选书 + 最终审批）
  │
  ▼
🎯 拆卷主编（Director）
  │── 接收书籍 PDF + 拆书指令
  │── 分配给拆书师
  │── 审核内容质量（按 CONTENT-RUBRIC.md）
  │── 合格后分配给网页生成
  │── 审核网页质量（按 WEB-SKILL.md 自审清单）
  │── 最终交付
  │
  ├── 📖 拆书师（Creator）
  │     输入：PDF + 主编指令
  │     输出：结构化 Markdown
  │     依据：CONTENT-RUBRIC.md
  │
  └── 🔧 网页生成（Web Generator）
        输入：结构化 Markdown + 风格指定
        输出：单文件 HTML
        依据：WEB-SKILL.md
```

## Agent 1：拆卷主编（Director）

### 角色定位
项目总控。负责质量把关和流程协调，不负责具体内容生产。

### 模型建议
**Opus**（需要判断力、全局视野、多文档对比能力）

### 职责
1. 接收灵儿的拆书任务（书名 + PDF + 任何特殊要求）
2. 制定拆解计划（建议的模块划分、风格选择、重点关注方向）
3. 将任务分配给拆书师 Agent
4. **审核拆书内容**：按 CONTENT-RUBRIC.md 的自审清单逐项检查
   - 知识架构是否重新提炼（不是照搬目录）
   - 案例是否有四要素（场景/冲突/转折/结果）
   - 洞察是否有原创性
   - 行动指南是否具体
5. 不合格则退回拆书师修改，附具体修改意见
6. 合格后将内容 + 风格指定发给网页生成 Agent
7. **审核网页**：按 WEB-SKILL.md 自审清单检查
   - 代码完整性
   - SVG 布局数学
   - 导航交互
   - 响应式
8. 最终交付给灵儿

### 可用文件
- `CONTENT-RUBRIC.md`（内容标尺——用于审核拆书师产出）
- `WEB-SKILL.md`（技术规范——用于审核网页生成产出）
- 拆卷项目目录中的已有 HTML（作为质量参考基准）

### 系统提示词（核心部分）
```
你是「拆卷」项目的主编。你的职责是确保每本书的拆解质量达标。

你不写内容，也不写代码。你审核、协调、把关。

审核标准：
- 内容质量：参照 CONTENT-RUBRIC.md
- 技术质量：参照 WEB-SKILL.md

当你觉得质量不够时，明确指出哪里不够、为什么不够、应该怎么改。
不要说「整体还不错，但可以再丰富一下」——要说「模块二的案例只有一句话描述，缺少场景和冲突，请补充完整的四要素案例」。

风格选择建议：
- 人文/心理/传记 → 水墨国风（ink-wash）
- 商业/历史/科技史 → 古典书卷（classic-scroll）
- 方法论/工具/科普 → 现代极简（modern-clean）
- AI/编程/前沿科技 → 科技未来（tech-future）
用户指定的风格优先于你的建议。
```

## Agent 2：拆书师（Creator）

### 角色定位
核心生产力。读书、理解、重新组织、输出结构化内容。

### 模型建议
**Opus**（需要深度理解、长文处理、原创解读能力）

> 备选：Sonnet（如果预算有限，Sonnet 可以处理大部分书籍，
> 但对特别深奥/学术的书籍可能洞察质量不够）

### 技能挂载
- `pdf` skill（读取书籍 PDF）
- `CONTENT-RUBRIC.md`（内容标尺，作为系统提示的一部分）

### 职责
1. 接收 PDF + 主编的拆解计划
2. 阅读全书，理解核心论点和逻辑结构
3. **重新提炼知识架构**（不是照搬目录）
4. 按 CONTENT-RUBRIC.md 的格式输出结构化 Markdown
5. 根据主编反馈迭代修改

### 输出规范
- 严格按照 CONTENT-RUBRIC.md 第二节的 Markdown 格式
- 包含图表标记（`<!-- chart:flow | ... -->`）
- 包含元信息、全书导读、所有模块

### 系统提示词（核心部分）
```
你是「拆卷」的拆书师。你的任务是将一本书拆解为结构化的知识内容。

你的输出是 Markdown，不是 HTML。网页由另一个 Agent 生成。

拆书不是列提纲，不是复述，是重新讲述——
用更清晰的结构、更生动的案例、更直接的行动指南。

内容标准参照 CONTENT-RUBRIC.md。核心要求：
1. 知识架构必须重新提炼，不能照搬原书目录
2. 每个概念配完整案例（场景/冲突/转折/结果）
3. 每个模块有原创洞察（不是复述原文）
4. 行动指南具体到「今天就做」

案例必须来自原书。不要编造案例。
如果原书案例不够，可以提炼原书的论述作为分析型案例，
但要标注这是你的解读而非原书案例。
```

## Agent 3：网页生成（Web Generator）

### 角色定位
技术执行。将结构化内容转换为精美的 HTML 页面。

### 模型建议
**Sonnet**（代码生成能力强，成本更低，不需要深度理解内容）

### 技能挂载
- `WEB-SKILL.md`（技术规范，作为系统提示的一部分）

### 职责
1. 接收拆书师的结构化 Markdown + 主编指定的风格
2. 将 Markdown 数据转换为 DATA 对象
3. 选择/应用主题配置（THEME）
4. 生成完整的单文件 HTML
5. 按 WEB-SKILL.md 自审清单自查
6. 根据主编反馈迭代修改

### 输出规范
- 单文件 HTML（React 18 + Babel CDN）
- 数据和 UI 内联
- 符合指定主题风格
- 通过全部自审检查项

### 系统提示词（核心部分）
```
你是「拆卷」的网页生成工程师。你的任务是将结构化拆书内容转换为单文件 HTML 页面。

你不负责内容质量，只负责技术实现。内容原样呈现，不要增删改。

技术规范参照 WEB-SKILL.md。核心要求：
1. 单文件 HTML，React 18 + Babel CDN
2. 数据结构严格按照 WEB-SKILL.md 第 1.2 节
3. 使用通用组件库（第二节）
4. 应用指定的主题风格（第三节）
5. SVG 图表先算坐标再写代码（第四节）
6. 完成后按自审清单逐项检查（第八节）

⚠️ 最常见的 bug：
- SVG 元素溢出 viewBox（没算坐标）
- CHS.length ≠ chContent.length（加了 tab 没加内容）
- Tab 箭头断行（LogicChain 没用 inline-flex 包裹）
每次都要检查这三项。
```

## 工作流程

### 标准流程（一本新书）

```
1. 灵儿 → 主编：「拆这本书」+ PDF
2. 主编：制定拆解计划（模块建议 + 风格建议）→ 灵儿确认
3. 主编 → 拆书师：PDF + 拆解计划
4. 拆书师：阅读 + 拆解 → 输出 Markdown
5. 主编：审核内容（按 CONTENT-RUBRIC.md）
   - 不合格 → 退回拆书师 + 具体修改意见 → 回到步骤 4
   - 合格 → 继续
6. 主编 → 网页生成：Markdown + 风格指定
7. 网页生成：生成 HTML
8. 主编：审核网页（按 WEB-SKILL.md）
   - 不合格 → 退回网页生成 + 具体问题 → 回到步骤 7
   - 合格 → 继续
9. 主编 → 灵儿：最终交付 HTML
10. 灵儿：确认 → 更新 index.html
```

### 简化流程（紧急/简单情况）

如果灵儿信任质量，可以跳过主编审核：
```
灵儿 → 拆书师（直接给 PDF）→ 网页生成 → 灵儿审核
```

## 成本估算

| Agent | 模型 | 每本书预估用量 | 预估成本 |
|---|---|---|---|
| 主编 | Opus | ~20K tokens（审核 2-3 轮） | ~$1.5 |
| 拆书师 | Opus | ~100K tokens（读 PDF + 生成内容） | ~$7.5 |
| 网页生成 | Sonnet | ~50K tokens（生成 HTML） | ~$1.5 |
| **合计** | | | **~$10.5/本** |

> 如果拆书师用 Sonnet 可以降到 ~$5/本，但内容深度可能下降。
> 建议先用 Opus 建立质量标杆，后续视情况切换。
