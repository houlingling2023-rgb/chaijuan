# 拆书页面生成技能

本技能指导你将一本书拆解为高质量的单文件 HTML 阅读页面。以下所有原则来自真实迭代经验——每一条都对应一个曾经犯过的错误或用户反馈。

## 一、核心设计理念

拆书不是列提纲。读者来看拆解，是因为他们不想（或还没时间）读原书，但又想真正理解这本书在说什么。所以拆解要做到三件事：

1. **让人看清结构** — 每个概念都配图表（流程图、对比图、结构图、时间线），让抽象关系变成视觉可见的
2. **让人感受温度** — 每个概念都配详细案例，有人物、有场景、有冲突、有转折，不是"举个例子"一笔带过
3. **让人知道自己在哪** — 全书导读 + 每章逻辑链是标配，读者任何时候都能鸟瞰全局

这三样东西（图表、案例、导航）不是锦上添花，是及格线。初版就应该全部到位。

## 二、阅读架构（必须包含）

### 2.1 全书导读页（第一个 tab）

读者打开页面的第一个念头是"这本书讲什么、逻辑是什么"，不是直接跳进第一章。所以：

- 全书导读必须作为第一个 tab，在所有章节之前
- 包含一张**全书逻辑架构图**（SVG），鸟瞰所有章节的关系和递进逻辑
- 包含**章节速览卡片**，每章一个核心问题 + 一句话核心观点，可点击跳转
- 包含**阅读建议**，说明各章之间的呼应关系

### 2.2 每章逻辑链（每章开头）

每章正文开始前，紧跟在章节标题之后，放一条逻辑链（LogicChain），用箭头串联该章的推理脉络。让读者在深入之前先看到这一章的思维路线图。

格式：`核心问题 → 概念A(解释) → 概念B(解释) → … → 结论`

### 2.3 Tab 导航

- 第一个 tab 是"全书导读"，与后续章节 tab 之间用视觉分隔（如竖线）区分
- Tab 数量可能较多（6-8个），必须：
  - 设置 `flexShrink:0` 防止压缩
  - 隐藏原生滚动条，加右侧渐变遮罩提示"还有更多"
  - 紧凑化 padding（10px 14px 左右），字号控制在 13px

## 三、内容丰富度标准

### 3.1 图表

每个核心概念至少配一个图表。可用类型：

- `InkFlow` — 流程图，适合展示步骤/路径
- `InkCompare` — 对比图，适合展示两种思维/模式的区别
- `InkStructure` — 结构图，适合展示多维度框架
- `InkTimeline` — 时间线，适合展示阶段性发展
- `LogicChain` — 逻辑链，适合展示推理脉络

### 3.2 案例

案例不是"比如有个人…"一句话带过。好的案例包含：

- 具体人物和场景（"小明是一个工作三年的产品经理，最近被调到了一个全新的项目组…"）
- 冲突和困境（他遇到了什么问题，为什么用旧方法行不通）
- 转折点（什么时候意识到需要改变，或者某个概念怎样帮到了他）
- 结果和启发

每章至少 3-5 个详细案例。

## 四、SVG 图表技术规范

这是最容易出 bug 的地方。以下每条规则都对应一个真实的渲染错误：

### 4.1 布局数学：先算再写

**在写任何 SVG 代码之前，必须先用代码计算所有元素的坐标和边界。**

```
举例：5 个等宽节点，每个宽 w，间距 gap，viewBox 宽 V
totalWidth = 5*w + 4*gap
startX = (V - totalWidth) / 2
每个节点 x = startX + i * (w + gap)
最右边界 = startX + 4*(w+gap) + w

如果最右边界 > V → 溢出！必须缩小 w 或 gap 或增大 V
```

**教训**：曾经 viewBox 设为 700 但最右节点到了 772，导致最后一个章节被裁切。这种错误纯粹是没有提前计算坐标。

### 4.2 对齐原则

- 同层级节点的 y 坐标必须统一，不要用不规则的偏移"制造美感"，实际效果只会显得不规整
- 节点间连线使用水平直线（同一 y 轴上），不要斜线，视觉更清晰
- 所有文字元素加 `textAnchor="middle"` 居中对齐

### 4.3 交互

- 可点击的 SVG 节点：`<g>` 加 `style={{cursor:"pointer"}}` + `onClick`
- 内部文字加 `style={{pointerEvents:"none"}}` 确保点击穿透到 `<g>`

### 4.4 概念显示

- 每个节点下方的关键词区域，高度要能容纳实际内容（至少显示 4 个关键词）
- 超出部分用 "+N个" 标注

## 五、组件联动一致性

添加新组件或新 tab 时，必须同步更新所有引用点：

**教训**：曾经在 CHS 数组里加了"全书导读"（变成 6 项），但忘了同步更新 App 的 chContent 数组（仍然 5 项），导致 tab 和内容对不上，页面崩溃。

检查清单：
- CHS 数组的长度 = chContent 数组的长度
- 新组件的 props（如 onJump）要在 App 层正确传递
- PageNav 的 total、ReadBar 的 progress 都依赖 CHS.length，确认它们能自动适配

## 六、响应式与断行

### 6.1 LogicChain 箭头断行

逻辑链用 flexWrap 换行时，步骤和箭头必须包在同一个 inline-flex 容器里，防止箭头单独掉到行首。

```jsx
// 错误：箭头可能孤立在行首
<span>步骤A</span>
<span>→</span>
<span>步骤B</span>

// 正确：步骤和箭头绑定在一起
<span style={{display:"inline-flex",alignItems:"center",gap:4}}>
  <span>步骤A</span>
  <span>→</span>
</span>
```

### 6.2 Tab 栏溢出

Tab 数量 ≥ 5 时，横向空间不够。必须：
- 外层 `overflowX:"auto"` + 隐藏滚动条
- 右侧加渐变遮罩暗示可滚动
- Tab 设 `flexShrink:0`

## 七、合规与版权

拆书天然涉及版权。免责声明不是"想起来再加"，而是模板的一部分。

每个拆书页面底部必须包含：
```
⚠ 免责声明：本页面内容仅供个人学习与读书笔记整理之用，不用于任何商业目的。
书籍版权归原作者及出版社所有。如有侵权，请联系博主，将第一时间删除处理。
```

首页（index.html）底部也必须有类似的全站声明。

## 八、自审清单

**每次生成完整页面后，必须按以下清单逐项检查，并将结果展示给用户：**

### A. 代码完整性
- [ ] 括号平衡：用 Python 脚本计算 `{}`、`()`、`[]`、反引号的数量，差值必须为 0
- [ ] 组件引用一致：CHS.length === chContent.length
- [ ] 所有新组件的 props 都在父组件中正确传递
- [ ] Script 标签开闭匹配

### B. 布局数学
- [ ] 所有 SVG：计算每个元素的实际坐标，确认无溢出 viewBox
- [ ] 所有 SVG：同层级节点 y 坐标一致（除非有明确的设计理由）
- [ ] Flex/Grid 布局：在最小预期宽度（如 375px）下不会导致关键内容被裁切

### C. 内容完整性
- [ ] 全书导读页存在且为第一个 tab
- [ ] 每章开头有 LogicChain
- [ ] 每个核心概念有至少一个配图
- [ ] 每章有至少 3 个详细案例（有人物、场景、冲突、转折）
- [ ] 版权免责声明存在于页面底部

### D. 交互体验
- [ ] SVG 中可点击的节点有 cursor:pointer 和 onClick
- [ ] Tab 数量 ≥ 5 时有滚动遮罩提示
- [ ] LogicChain 的箭头不会在换行时孤立
- [ ] 导读 tab 与章节 tab 有视觉区分
- [ ] 章节速览卡片可点击跳转到对应章节

### E. 读者视角模拟
完成技术检查后，切换到读者视角回答以下问题：
- 打开页面，我第一眼能理解这本书讲什么吗？
- 全书架构图能完整显示吗？信息层次清晰吗？
- 每章的逻辑脉络一目了然吗？
- 案例读起来有代入感吗，还是像在背书？
- 有没有哪里让我想"这是什么意思"却找不到解释的？
